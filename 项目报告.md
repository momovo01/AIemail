## 前言

随着人工智能技术的快速发展，大语言模型（Large Language Model, LLM）在自然语言处理领域展现出卓越的能力，为日常办公自动化带来了新的可能性。电子邮件作为现代工作场景中最主要的沟通方式之一，每天占用大量时间进行处理。然而，许多邮件具有重复性、模板化的特征，完全可以通过智能化手段辅助处理。

本文设计并实现了一款基于 DeepSeek 大模型的邮箱回复智能体系统。该系统通过 IMAP/SMTP 协议与 QQ 邮箱交互，调用 DeepSeek API 实现邮件内容的智能分析与回复生成，并引入"人工审核后发送"的人机协同机制，在提升效率的同时确保回复质量与安全性。全文将从需求分析、系统设计、详细实现、测试评估等角度对该系统进行全面阐述。

## 1 系统需求分析与总体设计

### 1.1 项目背景与目标

在日常工作和生活中，电子邮件处理是一项高频但低效的任务。用户往往需要重复性地回复问候、确认事项、回答常见问题等，这些场景具有内容相似、结构固定的特点，非常适合引入 AI 辅助以减轻人工负担。然而，完全自动化的邮件回复存在误回复、语气不当、信息失真等风险。因此，本项目的目标是设计一款"AI 生成 + 人工把关"的半自动化邮件智能体，在追求效率的同时保留人的最终决策权。

具体目标包括：（1）自动监听指定邮箱的未读邮件；（2）利用大模型分析邮件意图并生成回复草稿；（3）提供简洁直观的审核机制，在用户确认后发送；（4）支持白名单过滤，仅处理可信发件人，降低安全风险。

### 1.2 邮箱回复智能体的功能需求

#### 1.2.1 核心功能

本系统的核心功能围绕"监听—分析—生成—审核—发送"五步流程展开。

（1）**邮件自动读取**：通过 IMAP 协议周期性连接邮箱服务器，自动检测并获取所有未读邮件，解析邮件头信息（发件人、主题、日期）及正文内容，同时支持 Multipart 格式（纯文本与 HTML）的邮件解析。

（2）**发件人白名单过滤**：系统维护一份 JSON 格式的发件人白名单，仅对白名单内的发件人进行回复处理。非白名单发件人的邮件将被自动跳过并记录日志，以此规避垃圾邮件、广告邮件带来的无效处理。

（3）**AI 智能分析与回复生成**：调用 DeepSeek 大模型 API 对邮件内容进行两阶段处理——第一阶段分析邮件意图、提取关键信息并评估紧急程度；第二阶段根据分析结果生成符合设定风格的回复文本。

（4）**人工审核与发送**：系统将 AI 生成的回复保存为本地草稿文件，同时在终端展示完整内容供用户审阅。用户可通过交互式命令选择"发送""跳过"或"退出"，确认发送后系统通过 SMTP 协议将回复发出并标记原邮件为已读。

#### 1.2.2 扩展功能

在核心功能之外，系统还实现了以下扩展功能以提升可用性。

（1）**多运行模式**：系统支持三种运行模式——`check` 模式执行单次检查并处理未读邮件，适用于即时处理场景；`run` 模式以可配置的时间间隔持续轮询（默认 300 秒），适用于后台长期运行；`review` 模式独立审核本地草稿箱中的待发回复，适用于延迟决策场景。

（2）**完整日志记录**：使用 Python logging 模块实现双通道日志输出，同时将日志写入文件和控制台，记录内容包括邮件处理流程、API 调用状态、发送结果等关键信息，便于故障排查与运行审计。

（3）**草稿持久化**：所有 AI 生成的回复以结构化文本文件的形式保存在本地 `drafts/` 目录中，文件名包含时间戳和收件人信息，支持随时回溯和重新审核。

（4）**Windows 终端兼容**：通过 `sys.stdout.reconfigure` 将标准输出编码设置为 UTF-8，解决 Windows 系统 GBK 编码下 emoji 字符导致的 UnicodeEncodeError 问题，确保跨平台正常运行。

### 1.3 技术选型与架构

#### （1）大模型选择

本系统选择 DeepSeek 作为底层大模型，主要基于以下考量：DeepSeek 是由深度求索公司开发的大语言模型，其 API 完全兼容 OpenAI SDK 接口规范，开发者可直接使用 `openai` Python 库进行调用，无需额外适配；同时 DeepSeek 在中文场景下表现优异，生成文本自然流畅，能够满足邮件回复对语言质量的高要求。DeepSeek 的定价策略具有较高性价比，适合个人用户和小型团队长期使用。本系统使用 `deepseek-chat` 模型，分析阶段采用较低温度（0.3）以保证输出的确定性和结构化，回复生成阶段采用较高温度（0.7）以保证文本的自然度和多样性。

#### （2）邮件协议

系统选用 IMAP（Internet Message Access Protocol）协议进行邮件接收，SMTP（Simple Mail Transfer Protocol）协议进行邮件发送。IMAP 相比 POP3 的优势在于支持在服务器端管理邮件状态（如标记已读、移动到文件夹），且可在多设备间同步状态，适合需要持久化管理的应用场景。QQ 邮箱的 IMAP 服务使用 993 端口建立 SSL 加密连接，SMTP 服务使用 465 端口建立 SSL 加密连接，确保数据传输安全性。需要注意的是，QQ 邮箱要求使用独立的"授权码"而非 QQ 密码进行第三方客户端登录，这是腾讯为增强账户安全设置的机制。

#### （3）开发环境

系统基于 Python 3.12 开发，充分利用 Python 生态中成熟的邮件处理和 AI 调用库。依赖管理方面，仅引入两个外部库：`openai`（用于 DeepSeek API 调用）和 `python-dotenv`（用于环境变量管理）。邮件收发功能完全基于 Python 标准库中的 `imaplib`、`smtplib` 和 `email` 模块实现，尽可能减少了外部依赖，降低了部署复杂度。敏感信息（API 密钥、邮箱密码）通过 `.env` 文件管理，确保不会意外泄露到版本控制系统中。

## 2 详细设计与实现

### 2.1 邮件收发模块设计

#### 2.1.1 接收邮件

邮件接收功能封装在 `email_client.py` 模块中，核心函数为 `fetch_unread_emails()`。该函数通过 `imaplib.IMAP4_SSL` 建立与 QQ 邮箱 IMAP 服务器的加密连接，调用 `login()` 方法完成身份认证，随后使用 `select("INBOX")` 选择收件箱，通过 `search(None, "UNSEEN")` 检索所有未读邮件。

获取到邮件 ID 列表后，函数逐封调用 `fetch(eid, "(RFC822)")` 获取完整的 RFC 822 格式邮件原始数据。每次请求的响应被封装为 `Email` 数据类对象，包含五个字段：`uid`（邮件唯一标识符，用于后续标记已读操作）、`sender_name` 和 `sender_email`（通过 `email.utils.parseaddr` 解析发件人头信息）、`subject`（解码后的邮件主题）、`body`（解析后的纯文本正文）以及 `date`（邮件发送时间）。

出于资源管理的考虑，所有 IMAP 请求均采用 `try...finally` 结构，在操作完成后确保调用 `logout()` 断开连接，避免在服务器端遗留大量空闲连接。

#### 2.1.2 解析邮件内容

邮件头解析方面，RFC 2047 标准允许邮件头使用 `=?charset?encoding?text?=` 格式对非 ASCII 字符进行编码。系统通过 `email.header.decode_header()` 函数解码此类编码字符串，并处理了分段解码和字符集回退（默认为 UTF-8）的边界情况。

邮件正文解析需要应对 Multipart 格式。`_parse_email_body()` 函数首先判断邮件是否为 `multipart` 类型，若是则递归遍历各个 part：跳过带有 `Content-Disposition: attachment` 标记的附件部分，优先收集 `text/plain` 类型的纯文本内容，其次收集 `text/html` 类型的富文本内容。对于 HTML 格式的邮件，系统使用正则表达式去除 `<style>` 标签和所有 HTML 标签，将其转换为可读的纯文本。对于非 multipart 邮件，直接尝试解码 payload 内容。这一设计保证了系统能够处理绝大多数邮件格式，具有良好的鲁棒性。

### 2.2 AI 回复生成模块设计

#### 2.2.1 Prompt 设计与优化

Prompt（提示词）工程是本系统的核心设计环节之一，直接影响 AI 生成回复的质量和风格一致性。本系统采用单一 System Prompt 统一设定角色、风格和约束，确保分析与回复两个阶段的行为一致性。

System Prompt 的设计遵循以下原则：（1）**角色明确化**——将 AI 定位为"友好亲切的邮件助手"而非通用聊天机器人，使其输出符合邮件场景的语境；（2）**风格具象化**——不仅给出"亲切"这样的抽象形容词，还提供了具体可操作的建议，如"适当使用'呀''哈''呢'等语气词"，使模型能够产生预期风格的输出；（3）**边界约束**——明确要求"不要透露这是 AI 生成的回复""不确定的信息要诚实说明"，防止模型产生幻觉内容。

在分析阶段的 user prompt 中，系统要求模型以 JSON 格式输出结构化分析结果（包含 intent、urgency、key_points 三个字段），这一设计实现了分析结果的可解析性，为将来可能的自动化决策提供了数据基础。在回复生成阶段，user prompt 仅提供邮件内容和基本指令，将风格控制完全交给 System Prompt，保证了生成回复的风格一致性。

温度参数方面，分析阶段使用 0.3 的低温以确保意图判断的准确性和 JSON 格式的正确性；回复生成阶段使用 0.7 的较高温度以增强文本的自然度，避免过于机械和重复。

#### 2.2.2 DeepSeek API 调用实现

AI 模块封装在 `ai_agent.py` 中，通过 OpenAI SDK 实例化客户端对象，配置 `api_key` 和 `base_url` 指向 DeepSeek 服务端点（`https://api.deepseek.com`）。由于 DeepSeek API 完全兼容 OpenAI 的 `/v1/chat/completions` 接口规范，可以直接使用 `client.chat.completions.create()` 方法进行调用，无需任何额外适配工作。

模块对外暴露两个高层函数：

- `analyze_email(email_body, subject, sender_name)`：调用 API 分析邮件意图，`max_tokens` 设置为 500（分析结果的预估长度上限），返回 AI 生成的 JSON 格式分析文本。
- `generate_reply(email_body, subject, sender_name)`：调用 API 生成回复文本，`max_tokens` 设置为 2000（回复邮件的合理长度上限），返回可直接发送的邮件正文。

两个函数均接受发件人名称、邮件主题和正文作为参数，将这些信息拼接进 user prompt 中传递给模型。这种设计使模型在生成回复时能够参考发件人身份和邮件主题，从而输出更具针对性的内容。

### 2.3 人机协同与审核模块

#### （1）日志记录功能

系统使用 Python `logging` 模块建立了完善的日志体系。日志级别设置为 INFO，格式包含精确到秒的时间戳和日志级别标签，同时输出到控制台（StreamHandler）和日志文件（FileHandler，文件名为 `email_agent.log`，UTF-8 编码）。关键操作均有日志记录：未读邮件数量的发现、白名单过滤的跳过说明、AI 分析结果、草稿保存路径、发送成功或失败的状态，以及用户的操作选择。这些日志既为运行监控提供了实时可见性，也为事后审计保留了完整记录。

#### （2）二次确认发送机制

发送确认机制采用了终端交互式审核的设计。在 `process_emails()` 函数中，每封邮件的 AI 回复生成后，系统首先将回复以结构化格式保存到本地 `drafts/` 目录的文本文件中，然后在终端以分隔线清晰展示发件人、主题和 AI 生成的完整回复文本，最后通过 `input()` 函数等待用户输入决策。

用户有三种选择：输入 `y` 触发 SMTP 发送流程并将原邮件标记为已读；输入 `n` 跳过当前邮件但保留草稿文件以备后审；输入 `q` 立即退出整个处理循环。在 `review` 模式下，系统读取已保存的草稿文件，按时间排序逐条展示，并提供"发送""删除草稿""下一条""退出"四种操作，使用户可以在任何方便的时间集中审核待发回复。

这一设计体现了"人机协同"的核心理念——AI 负责生成初稿，人类负责最终决策，双方各司其职，既大幅提升了效率，又为回复质量提供了安全保障。

### 3.3 性能评估

#### （1）响应速度

系统端到端处理单封邮件的耗时由以下环节组成：IMAP 连接与邮件检索（约 1-2 秒）、DeepSeek API 分析调用（约 1-2 秒）、DeepSeek API 回复生成调用（约 1-2 秒）、本地草稿保存（小于 0.1 秒）。在实际测试中，单封邮件从检查到草稿生成完成的总耗时约为 3-4 秒（不含用户审核决策时间）。这一响应速度满足个人日常使用的性能需求。

#### （2）成本估算

DeepSeek API 采用按 Token 计费模式。以每次处理估算：分析调用约消耗 200-400 输入 Token 和 100-150 输出 Token；回复生成调用约消耗 300-500 输入 Token 和 150-300 输出 Token。单封邮件的总 Token 消耗约在 750-1350 之间。按照 DeepSeek 官方定价（`deepseek-chat` 模型输入 1 元/百万 Token，输出 2 元/百万 Token），单封邮件的 API 调用成本约为 0.001-0.002 元人民币，以每天处理 50 封邮件计算，月度成本不超过 3 元，具有极高的成本效益。

## 4 创新点与不足

### 4.1 创新点

（1）**人机协同的审核机制**：区别于完全自动化的邮件机器人，本系统采用"生成—审核—发送"的协同模式，将 AI 的效率优势与人类的判断力有机结合。AI 负责理解邮件内容并起草回复，人类只需快速审阅确认，既避免了全自动发送的失控风险，又保留了半自动化带来的效率提升。草稿持久化设计更使用户可以在任何方便的时间集中审核，灵活度更高。

（2）**精细化 Prompt 工程**：系统将 AI 调用拆分为"分析"和"生成"两个独立阶段，分别采用不同的温度和输出格式约束。分析阶段强调结构化（JSON 格式）和准确性（低温），生成阶段强调自然度（高温）和风格一致性（System Prompt 统一控制）。这种"先理解、后回复"的两阶段设计相比一步到位的方案更加可控和可解释。

（3）**白名单安全过滤**：将回复范围限定在白名单发件人，从机制层面避免了对垃圾邮件、钓鱼邮件的误回复，是一种简单有效的安全策略。同时白名单采用独立的 JSON 配置文件，修改方便，不涉及代码变更。

（4）**极简架构与低依赖**：整个系统仅依赖两个外部库（openai、python-dotenv），邮件处理全部使用 Python 标准库实现，部署门槛极低。代码总量约 300 行，结构清晰，模块职责分明，易于理解和二次开发。

### 4.2 存在的问题与改进方向

（1）**轮询效率问题**：当前系统采用定时轮询机制，邮件到达与处理之间存在最长 `CHECK_INTERVAL` 的延迟。在需要实时响应的场景下，更为理想的方案是采用 Webhook 或 IMAP IDLE（实时推送）机制。IMAP IDLE 可以在邮件到达时立即收到服务器通知，实现近乎实时的响应。后续版本可引入 IDLE 支持作为轮询的补充方案。

（2）**上下文记忆缺失**：当前系统每次处理邮件均为独立会话，不保存与发件人的历史对话记录。这意味着 AI 在回复同一发件人的后续邮件时，无法参考以往的交流内容。引入对话记忆模块（如基于 SQLite 的轻量级邮件历史存储）可使回复更具连续性和个性化。

（3）**HTML 邮件回复**：当前系统仅生成纯文本格式的回复邮件。虽然纯文本足以应对大多数场景，但在商务场合中，带有适当格式的 HTML 邮件（如加粗、列表、链接等）往往更具专业感。未来可扩展支持 Markdown 转 HTML 的回复格式。

（4）**多语言支持**：当前 System Prompt 为中文设定，对英文邮件的处理能力受限于模型本身的多语言能力。可通过检测邮件语言并动态切换 Prompt 语种来实现更精准的多语言回复。

（5）**附件处理**：当前系统在解析邮件时会跳过附件部分。对于包含附件的邮件（如简历、合同等），若能提取附件信息并纳入分析上下文，将大幅扩展系统的适用场景。

（6）**敏感信息安全**：当前 API Key 和邮箱授权码以明文形式存储在 `.env` 文件中。对于更高安全要求的场景，可引入系统密钥环（如 Windows Credential Manager）或加密存储方案。

## 总结

本文设计并实现了一款基于 DeepSeek 大模型的邮箱回复智能体系统。该系统围绕"监听—分析—生成—审核—发送"的五步流程，通过 IMAP/SMTP 协议实现与 QQ 邮箱的交互，利用 DeepSeek API 完成邮件内容的智能分析和高品质回复生成，并通过白名单过滤和人工审核机制保障了系统的安全性与可靠性。

系统采用 Python 语言开发，代码架构简洁清晰，仅依赖两个外部库，总计约 300 行代码即实现了完整可用的邮件智能助手功能。实际测试表明，系统能够正确识别白名单发件人，准确分析邮件意图，生成风格友好、语言自然的回复文本，端到端处理单封邮件耗时约 3-4 秒，单封邮件的 API 调用成本约 0.001 元，具备良好的性能和极高的成本效益。

该系统的核心价值在于探索了"人机协同"在邮件处理场景中的实践路径——不是用 AI 完全替代人，而是让 AI 承担重复性劳动，让人专注于决策与判断。这一设计理念可推广至更多办公自动化场景，如文档起草、会议纪要、客户服务等，具有广泛的参考意义。未来将在实时性、上下文记忆、多语言支持和安全加密等方向持续改进。

## 附录 1：核心代码清单

**项目文件结构：**

```
AIemail/
├── main.py            # 主入口 + 轮询 + 审核（约 235 行）
├── config.py          # 配置管理（约 27 行）
├── email_client.py    # IMAP/SMTP 邮件收发（约 157 行）
├── ai_agent.py        # DeepSeek API 调用（约 71 行）
├── whitelist.json     # 发件人白名单
├── requirements.txt   # 依赖清单
├── .env.example       # 环境变量模板
├── drafts/            # 草稿保存目录
└── email_agent.log    # 运行日志
```

**配置管理模块（config.py）：**
使用 `python-dotenv` 加载 `.env` 文件，提取 DeepSeek API 密钥、邮箱地址与授权码、IMAP/SMTP 服务器地址、轮询间隔等配置项。对必要的环境变量进行非空校验，缺少时抛出 ValueError 并给出明确提示。

**邮件收发模块（email_client.py）：**
定义了 `Email` 数据类，实现了 `fetch_unread_emails()`（IMAP 获取未读邮件）、`mark_as_read()`（标记已读）、`send_email()`（SMTP 发送）四个对外接口。内部包含邮件头解码、Multipart 正文解析等辅助函数。关键设计点包括：使用 `try...finally` 确保 IMAP/SMTP 连接可靠关闭；优先提取纯文本正文，HTML 正文通过正则表达式去除标签作为回退方案；邮件主题和发件人名称均通过 `decode_header` 进行 RFC 2047 解码。

**AI 调用模块（ai_agent.py）：**
基于 OpenAI SDK 初始化指向 DeepSeek 的客户端，定义了统一的 System Prompt，实现了 `analyze_email()` 和 `generate_reply()` 两个函数。分析阶段使用 temperature=0.3 和 JSON 结构化输出，回复生成阶段使用 temperature=0.7 和纯文本输出。

**主入口模块（main.py）：**
实现了命令行参数路由、白名单加载、邮件处理主循环、草稿保存、终端交互审核、持续轮询和草稿审核等功能。日志同时输出至文件和控制台。`process_emails()` 函数将邮件收发、AI 调用、草稿保存和交互审核串联为完整的工作流。

## 附录 2：操作说明

**环境准备：**

1. 安装依赖：`pip install -r requirements.txt`
2. 创建 `.env` 文件并填入：DeepSeek API Key、QQ 邮箱地址、QQ 邮箱授权码
3. 编辑 `whitelist.json`，添加需要自动回复的发件人邮箱及名称

**运行方式：**

- `python main.py check` —— 单次检查并处理未读邮件，逐封审核
- `python main.py run` —— 持续轮询模式（默认每 300 秒检查一次），按 Ctrl+C 退出
- `python main.py review` —— 审核本地草稿箱中的历史待发回复
